# sdgo Design

This document records the current tool plan and maintenance rules.

## Positioning

`sdkitgo` is a template repository, not the daily CLI name for new projects.

`sdgo` is the CLI used to:

- create a new project from the `sdkitgo` template.
- normalize project names, module paths, imports, and command entry directories.
- run a generated project with built-in hot reload.
- generate module skeletons and matching docs.

The generated project should not keep `sdkitgo` as its application command name. `sdkitgo` is only the template source identity.

## Command Naming

The CLI binary is `sdgo`.

Reasons:

- short enough for daily use.
- avoids conflict with the generated project's own command.
- keeps `sdkitgo` available as the template repository name.

Examples:

```bash
sdgo new demo
sdgo dev
sdgo gen module user
```

## Project Generation Rules

`sdgo new <project>` uses `sdkitgo` as source and creates a new project directory.

Source resolution order:

1. `--source`
2. `SDKITGO_SOURCE`
3. current directory
4. nearby `sdkitgo`
5. `/Users/huwenlong/data/lab/sdkitgo`

The source must contain:

- `go.mod`
- `cmd/sdkitgo/main.go`
- `bootstrap`
- `command`

Rewrite rules:

- `go.mod`: first `module ...` line becomes the requested module path.
- Go files: imports from `sdkitgo/...` become `<module>/...`.
- command path: `cmd/sdkitgo` becomes `cmd/<normalized-project>`.
- command use name: `sdkitgo` becomes `<normalized-project>` in Go command metadata.
- config and deploy files: project identifiers in YAML, Makefile, Dockerfile, and Dockerfile.dockerignore are rewritten.

Name normalization:

- target directory uses the user input, for example `demo-app`.
- Go package/command directory uses a compact lowercase form, for example `demoapp`.
- snake case is used for generated module package names, for example `user-profile` becomes `user_profile`.

Skip rules:

- do not copy `.git`, `.gitnexus`, `.cache`, `.claude`, `.codex`, `.idea`, `.vscode`.
- do not copy `bin`, `dist`, `logs`, `storage`, `tests`, `tmp`, `.tmp`, `vendor`, `node_modules`.
- do not copy `.DS_Store`, `.air.toml`, `AGENTS.md`, `CLAUDE.md`, `tips.md`.
- do not follow symlinks.

## Hot Reload Rules

`sdgo dev` and `sdgo run` use the same runner.

Default command:

```bash
go run ./cmd/<project> serve
```

The default command is only inferred when exactly one `cmd/*/main.go` exists. If there are multiple command entries, pass `--cmd`.

Default watch roots are existing directories from:

- `app`
- `bootstrap`
- `command`
- `configs`
- `core`
- `modules`
- `pkg`
- `cmd`

The watcher polls file snapshots and restarts the child process when size or modtime changes.

Shutdown behavior:

- sends interrupt first.
- waits briefly.
- kills the process if it does not exit.

## Module Generation Rules

`sdgo gen module <name>` creates application module skeletons under `modules/<name>`.

It also creates docs:

- `docs/modules/<name>.md`
- `docs/usage/<name>.md`

Module docs are generated by default because new modules should carry usage and design notes from the start.

## Maintenance Rules

- Keep `sdgo` small and standard-library first.
- Do not add a file watcher dependency unless polling becomes a real bottleneck.
- Do not introduce compatibility for old project templates unless explicitly needed.
- Keep generation rules deterministic and easy to inspect.
- Prefer explicit rewrites over broad text replacement.
- Do not copy local runtime data into generated projects.
- Tests should cover both fake fixtures and real-template generation behavior where practical.

## Current Boundaries

`sdgo` does not:

- initialize git repositories.
- install Go dependencies for the generated project.
- run database or Redis services.
- edit the `sdkit` core repository.
- migrate existing projects from older layouts.

Those can be added later as separate commands if they become real workflow needs.
