# sdgo Design

This document records the current tool plan and maintenance rules.

## Positioning

`sdkitgo` is a template repository, not the daily CLI name for new projects.

`sdgo` is the CLI used to:

- create a new project from the `sdkitgo` template.
- normalize project names, module paths, imports, and command entry directories.
- run a generated project with built-in hot reload.
- upgrade the installed `sdgo` CLI binary.

The generated project should not keep `sdkitgo` as its application command name. `sdkitgo` is only the template source identity.

## Command Naming

The CLI binary is `sdgo`.

Reasons:

- short enough for daily use.
- avoids conflict with the generated project's own command.
- keeps `sdkitgo` available as the template repository name.

Examples:

```bash
sdgo new demo
sdgo dev
sdgo upgrade
```

## Project Generation Rules

`sdgo new <project>` uses `sdkitgo` as source and creates a new project directory.

Source resolution order:

1. `--source`
2. `SDGO_TEMPLATE_<NAME>_SOURCE`
3. `SDKITGO_SOURCE` for the default `sdkitgo` template
4. `SDGO_TEMPLATE_SOURCE`
5. current directory
6. nearby template directory
7. built-in template source list

`--source` and `SDKITGO_SOURCE` may be local directories or Git URLs. Git URL sources are cloned into a temporary directory with:

```bash
git clone --depth=1 <source>
```

When `--branch` is provided, it is cloned with:

```bash
git clone --depth=1 --branch <branch> --single-branch <source>
```

Private repositories are supported as long as the user's local `git` command can authenticate.

Templates are selected with `--template`. The default template is `sdkitgo`, whose built-in source is `git@gitee.com:sd0/sdkitgo.git`. Future templates can be built into the CLI by adding their default source to `builtinTemplateSources`, while still allowing users to override with `--source` or `SDGO_TEMPLATE_<NAME>_SOURCE`.

Go template sources must contain:

- `go.mod`
- exactly one `cmd/*/main.go`
- `bootstrap`
- `command`

Node/frontend template sources must contain:

- `package.json`

Go rewrite rules:

- `go.mod`: first `module ...` line becomes the requested module path.
- Go files: imports from the template module path become `<module>/...`.
- command path: `cmd/<template-command>` becomes `cmd/<normalized-project>`.
- command use name: the template command name becomes `<normalized-project>` in Go command metadata.
- config and deploy files: project identifiers in YAML, Makefile, Dockerfile, and Dockerfile.dockerignore are rewritten.

Node rewrite rules:

- `package.json`: `name` becomes the project directory name.
- `package-lock.json`: root package name is rewritten when present.
- `index.html` and Markdown files: the template package name is rewritten to the project directory name.

Name normalization:

- target directory uses the user input, for example `demo-app`.
- Go package/command directory uses a compact lowercase form, for example `demoapp`.
- snake case is used for generated module package names, for example `user-profile` becomes `user_profile`.

Skip rules:

- do not copy `.git`, `.gitnexus`, `.cache`, `.claude`, `.codex`, `.idea`, `.vscode`.
- do not copy `bin`, `build`, `coverage`, `dist`, `logs`, `storage`, `tests`, `tmp`, `.tmp`, `vendor`, `node_modules`.
- do not copy common frontend cache/build directories such as `.next`, `.nuxt`, `.svelte-kit`, and `.vite`.
- do not copy `.DS_Store`, `.air.toml`, `AGENTS.md`, `CLAUDE.md`, `tips.md`.
- do not follow symlinks.

## Hot Reload Rules

`sdgo dev` and `sdgo run` use the same runner.

Default command:

```bash
go run ./cmd/<project> serve
```

The default command is only inferred when exactly one `cmd/*/main.go` exists. If there are multiple command entries, pass `--cmd`.

The default watch root is the current project directory. The watcher skips generated, runtime, dependency, VCS, and editor/cache directories, then restarts the child process when watched file size or modtime changes.

Go test files (`*_test.go`) are ignored because changing tests should not restart the development server.

Shutdown behavior:

- sends interrupt first.
- waits briefly.
- kills the process if it does not exit.

File watching is enabled by default because `sdgo` is primarily a development tool. Use `--no-watch` when running under a process manager or production-style environment.

## Reserved Module Generation Plan

Module generation is intentionally not exposed in the CLI yet. The current reserved plan is to create application module skeletons under `modules/<name>`.

It also creates docs:

- `docs/modules/<name>.md`
- `docs/usage/<name>.md`

Module docs are generated by default because new modules should carry usage and design notes from the start.

## CLI Upgrade Rules

`sdgo upgrade` upgrades the CLI itself by delegating to Go's standard installer:

```bash
go install github.com/huwenlong92/sdgo/cmd/sdgo@latest
```

The command accepts an optional version argument, for example:

```bash
sdgo upgrade v0.2.0
```

Project template upgrades are intentionally separate and are not handled by this command.

## Maintenance Rules

- Keep `sdgo` small and standard-library first.
- Do not add a file watcher dependency unless polling becomes a real bottleneck.
- Do not introduce compatibility for old project templates unless explicitly needed.
- Keep generation rules deterministic and easy to inspect.
- Prefer explicit rewrites over broad text replacement.
- Do not copy local runtime data into generated projects.
- Tests should cover both fake fixtures and real-template generation behavior where practical.

## Current Boundaries

`sdgo` does not:

- initialize git repositories.
- install Go dependencies for the generated project.
- run database or Redis services.
- edit the `sdkit` core repository.
- migrate existing projects from older layouts.

Those can be added later as separate commands if they become real workflow needs.
